## 一、项目简介

ModernBlogPanel 是一款基于 Python + Flask 的静态博客生成系统，内置 Web 后台管理界面，无需数据库，无需服务器端渲染，最终产物为纯静态 HTML 文件，可直接部署到 GitHub Pages、Cloudflare Pages 等任意静态托管平台，无需像hexo那般面对黑乎乎的终端，只需简单几步即可获得动态博客般的后台管理体验，小白轻松上手。

核心特性如下：

- 双语支持：中文（zh）与英文（en）双语站点，共用同一套主题模板，自动生成各自独立的页面与索引。
- Markdown 写作：全部文章和页面均使用 Markdown 格式编写，支持代码高亮、表格、目录、围栏代码块等扩展语法。
- 图形化后台：通过浏览器访问本地后台，完成文章新建、编辑、删除、构建、推送一条龙操作，无需手动操作命令行。
- 一键部署：填写 GitHub 仓库地址后，点击"推送"按钮即可将构建产物强制推送至 GitHub Pages，内置部署互斥锁，避免重复点击导致 git 锁文件冲突。
- 三级封面图机制：单篇文章 frontmatter `cover` 字段 > 全局固定图池随机抽取 > 随机图 API（带防撞图 uid 参数），逐级兜底，最后一级如三者均未配置则显示扁平化图标占位，零构建期网络开销。
- 内容组织：支持分类、标签、年月归档三种维度，首页侧栏可按维度筛选并支持叠加筛选，构建期服务端渲染，SEO 友好。
- 前台个性化：访客可通过左下角设置面板自定义侧边栏位置和主题色（存于浏览器本地，不影响其他访客），站长也可在后台设置全站默认主题色、首页每页文章数、多个动效开关。
- 备份与恢复：内置本地一键打包备份、恢复前自动快照兜底，并可对接自建的云端 PHP 备份接口做异地备份。
- 性能优化：首屏背景图自动添加 `fetchpriority="high"`，优化 LCP（最大内容绘制）指标。
- SEO 友好：自动生成 sitemap.xml、robots.txt，内置 Open Graph 标签、JSON-LD 结构化数据、hreflang 多语言声明。
- Cloudflare 集成：支持在后台直接触发 Cloudflare 全站缓存清理，无需登录 CF 控制台。

## 二、环境要求与安装

### 系统要求

- Python 3.10 或更高版本
- Git（用于推送至 GitHub）
- 操作系统：Windows、macOS、Linux 均支持

### 安装依赖

先git本项目源码至本地

```bash
git clone https://github.com/Cnfte/ModernBlog.git
```
在项目根目录执行以下命令安装所需 Python 库：

```bash
pip install -r requirements.txt
```
### 文件结构

确保以下文件位于同一目录下：

```
项目根目录/
├── main.py            后台管理服务
├── builder.py         静态站点构建引擎
├── backup_manager.py  备份模块（本地打包/恢复 + 云端接口对接，可选，缺失时后台会隐藏备份相关报错并禁用该功能）
└── themes/
    └── default/
        ├── index.html      首页模板
        ├── post.html       文章页模板
        └── _enhance.html   前台设置面板 / 分类标签归档 / 动效等共享片段，被 index.html 和 post.html 通过 include 引入
```

## 三、启动后台管理面板

在项目根目录执行：

```bash
python main.py
```

启动成功后，终端会输出以下信息：

```
=======================================================
  INDEX // MOE_SYSTEM v8.0.0
  Flask WebUI Edition
  访问地址: http://0.0.0.0:32323
  本机访问: http://127.0.0.1:32323
=======================================================
```

用浏览器打开 `http://127.0.0.1:32323` 即可进入后台管理界面。

首次运行时，程序会自动完成以下初始化操作：

- 创建 `content/posts/zh`、`content/posts/en` 文章目录
- 创建 `content/pages/zh`、`content/pages/en` 页面目录
- 创建 `content/attachments` 素材目录
- 生成默认 `content/config.json` 配置文件

## 四、后台界面概览

后台顶部为标签导航栏，共分七个功能区：

| 标签 | 功能说明 |
|------|----------|
| 内容 | 编写、编辑、删除文章与页面，触发构建和部署 |
| 素材 | 上传图片等文件，获取可用于 Markdown 的链接 |
| 配置 | 填写站点名称、URL、社交账号、部署参数等全部配置项 |
| 追番 | 管理正在追看的动画列表，展示在站点侧边栏或专属页面 |
| 友链 | 管理友情链接，填写友站名称、网址和描述 |
| 备份 | 本地一键备份/恢复，以及对接自建云端备份接口 |
| 日志 | 实时查看构建、部署、缓存清理等操作的执行日志 |

## 五、配置项详解

所有配置项保存在 `content/config.json`，可在后台"配置"标签中通过图形界面修改，保存后立即写入文件。以下逐项说明每个配置字段的用途与填写规范。

---

### 基础站点

**site_name**
站点标题，显示在浏览器标签页标题、Open Graph 分享卡片以及 JSON-LD 结构化数据中。填写你的博客名称即可，例如 `「代码与星空」` 或 `Alex's Blog`。建议不超过 30 个字，避免在搜索引擎结果页被截断。

**site_url**
站点完整访问地址，必须包含协议头，例如 `https://username.github.io` 或 `https://blog.example.com`。该字段用于生成 sitemap.xml 中的绝对 URL、Canonical 标签、hreflang 声明以及 robots.txt 中的 Sitemap 地址。填写错误会导致搜索引擎收录异常，请务必确认地址与实际部署地址一致。如果使用自定义域名，填写自定义域名；如果是 GitHub Pages 默认域名，填写 `https://用户名.github.io`。

**logo_text**
导航栏左上角显示的 Logo 文字。通常填写站点缩写、英文名称或符号性文字，例如 `GEEK.LOG` 或 `STELLA`。该字段仅影响视觉展示，不影响 SEO。

**start_date**
建站日期，格式为 `YYYY-MM-DD`，例如 `2024-03-15`。该字段会显示在页脚或博主信息卡片中，用于展示站龄信息。填写实际开始运营博客的日期即可。

**hero_title**
首页 Hero 区域（顶部大图横幅）的主标题文字。通常是一句有个性的欢迎语或口号，例如 `WELCOME TO THE VOID` 或 `KEEP CODING, STAY CURIOUS`。支持大写英文风格，字数建议控制在 20 个字以内以保证视觉效果。

**hero_subtitle**
Hero 区域主标题下方的副标题文字。可以是对站点的一句话简介，或者配合主标题的补充语，例如 `一个程序员的日常记录` 或 `Code · Anime · Life`。

**site_keywords**
站点 SEO 关键词，多个关键词之间用英文逗号分隔，例如 `技术博客, 前端开发, Python, 动漫, 二次元`。这些关键词会写入 `<meta name="keywords">` 标签。现代搜索引擎对 keywords 标签的权重已大幅降低，但填写规范的关键词仍有助于部分搜索引擎的分类识别。建议填写 5 至 10 个与站点内容高度相关的词语。

**site_description**
站点 SEO 描述文字，一段简短介绍站点内容的话，建议 80 至 155 个字符之间。该内容会写入 `<meta name="description">` 标签，并作为 Open Graph 分享卡片的描述文字。这是影响搜索引擎点击率最重要的元数据之一，请认真撰写，清晰描述站点的主要内容与受众。例如：`一个专注于前端开发与动画文化的个人博客，分享技术教程、观影记录与生活随笔。`

**bg_url**
全站默认背景图片的 URL 地址。当文章没有单独设置背景图、固定图池及随机图 API 也均未配置时，将使用此处填写的图片作为最终兜底。可填写任意可公开访问的图片直链，建议使用宽高比接近 16:9 的横版图片，分辨率不低于 1920x1080。也可使用随机图片 API，例如 `https://picsum.photos/1920/1080`。首页/文章页的首屏背景图会自动加上 `fetchpriority="high"` 属性以优化 LCP 加载性能。

**hero_bg_url**
专门用于首页 Hero 区域的背景图 URL。如果留空，首页 Hero 区域将使用 `bg_url` 字段指定的图片。如果希望首页与文章页使用不同的背景图，在此处单独填写首页背景图地址即可。

**post_bg_urls**
文章封面图固定图池。在后台"配置"标签中每行填写一个图片直链 URL（保存时会自动按行/逗号拆分为列表）。构建时，每篇没有单独指定封面的文章会从列表中随机抽取一张，使不同文章在视觉上产生差异。若此列表为空，则回退使用下方的 `random_img_api`；若两者均为空，则最终回退到 `bg_url`。示例（三行）：

```
https://example.com/wallpaper/a.jpg
https://example.com/wallpaper/b.jpg
https://picsum.photos/1920/1080?random=3
```

单篇文章在编辑时可在"封面 URL"输入框中手动填写（对应 front matter `cover` 字段），优先级最高，会覆盖图池和随机图 API 的结果。

**random_img_api**
随机图片 API 地址，仅在 `post_bg_urls` 为空时生效（默认值为 `https://www.dmoe.cc/random.php`）。构建时不会发起任何网络请求，而是将该 API 地址拼接每篇文章独有的 `uid` 参数（例如 `https://www.dmoe.cc/random.php?v=174233xxxxxx`），由浏览器在访问时各自请求，天然防止不同文章出现相同背景图，且零构建开销。

**og_image**
Open Graph 封面图的 URL 地址。当博客链接被分享到微信、Twitter、Telegram、Discord 等平台时，会自动拉取此图片作为预览卡片的封面图。建议使用尺寸不小于 1200x630 像素的横版图片，图片中可包含站点 Logo 或名称。填写完整的 https 地址。

### 展示与动效

**posts_per_page**
首页每页显示的文章数量，默认 `9`。构建时写入页面，首页文章列表下方会据此生成分页条；切换分类/标签/归档筛选后再分页，不会重复计数。少于一页的文章数量时分页条不显示。

**theme_color**
全站默认主题色（十六进制颜色值，例如 `#ff99cc`），在后台"配置"标签中通过颜色选择器填写。构建期会直接推导出全套配色变量（辅助色、渐变、边框色等）写入页面 `:root`，保证访客第一次打开网站、还没碰过前台调色盘之前看到的就已经是这个颜色，包括页脚图标在内的所有元素都会跟随。访客仍可以通过前台左下角"显示设置"面板临时调整自己看到的主题色（只影响该访客本地浏览器，存储于 `localStorage`，不影响其他人）。

**enable_hero_typing**
布尔值，控制首页 Hero 区域副标题是否使用打字机效果逐字显示（速度随机、标点后额外停顿，模拟真人打字），默认关闭。这是站长的设计选择，不是访客可自行切换的前台偏好。

**enable_particles**
布尔值，控制访客鼠标移动时是否拖出彩色粒子跟随尾迹，默认开启。关闭后对应的 `<canvas>` 节点都不会生成，不占用额外渲染/计算资源。

**enable_icon_spin**
布尔值，控制设置齿轮图标的常态缓慢自转，以及侧边栏分类/标签/归档 widget 标题图标悬停时的旋转反馈，默认开启。

**enable_tag_bounce**
布尔值，控制点中标签/分类时是否有放大回弹的小动画，默认开启。

以上三个动效开关相互独立，可以任意组合开关；侧边栏位置、主题色、分类/标签/归档筛选、分页、回到顶部按钮这些功能性部分不受这几个开关影响，始终可用。

### 页脚内容

**footer_text**
页脚区域的自定义文字，支持 HTML 标签。例如可以填写版权声明、ICP 备案号、联系邮箱等。链接请使用 `href=` 属性，例如：

```html
本站由 GitHub Pages 强力驱动 · <a href="https://beian.miit.gov.cn" target="_blank">京ICP备XXXXXXXX号</a>
```

如果留空，系统会回退使用旧版的 `footer_custom` 字段内容（向后兼容旧配置文件）。

**footer_custom**
旧版页脚配置字段，仅在 `footer_text` 为空时生效，用于兼容早期配置文件。新用户直接使用 `footer_text` 即可，无需关注此字段。

### 弹窗公告与通知

**site_notice**
弹窗公告内容，支持 HTML。首次访问站点时会以弹窗形式展示给访客。适合用于发布重要通知、近期活动、站点迁移提醒等信息。链接请务必使用 `href=` 属性而非 `src=`，系统会在构建时自动将 `<a src="...">` 修复为 `<a href="...">`，但为了规范建议直接写正确的形式。如果不需要弹窗公告，留空即可。

**show_notice_widget**
布尔值，控制是否在侧边栏同时显示一个公告 Widget。勾选后，`site_notice` 中的内容不仅会在弹窗中出现，还会在文章页侧边栏固定展示一个小公告卡片。适合长期需要在侧边栏展示固定信息（如赞助渠道、重要链接）的场景。

### 博主信息

**username**
博主 ID 或昵称，显示在侧边栏博主信息卡片、文章元信息等位置。填写你在博客中希望展示的名字，例如 `Stellaria` 或 `老王`。

**avatar_url**
博主头像图片的 URL 地址。建议使用正方形或圆形裁剪的头像图片，尺寸不低于 200x200 像素。可使用 GitHub 头像地址（`https://github.com/用户名.png`）或其他图床地址。

**bio**
博主个人简介，一句话介绍自己，显示在侧边栏博主卡片中。例如 `热爱编程与动漫的独立开发者` 或 `Keep it simple. Keep it real.`，建议不超过 50 个字。

**email**
联系邮箱地址，例如 `me@example.com`。填写后会在博主信息卡片中生成邮件联系图标链接。如不希望公开邮箱，留空即可。

**github_url**
GitHub 主页完整 URL，例如 `https://github.com/username`。填写后会在社交图标栏生成 GitHub 链接。

**telegram_url**
Telegram 频道或个人主页的完整 URL，例如 `https://t.me/channel_name`。填写后生成 Telegram 图标链接。

**bilibili_url**
Bilibili 个人主页的完整 URL，例如 `https://space.bilibili.com/123456789`。面向中文受众的博主推荐填写。

**twitter_url**
Twitter（现更名为 X）个人主页完整 URL，例如 `https://twitter.com/username` 或 `https://x.com/username`。

**rss_url**
RSS 订阅地址。如果你的托管平台提供 RSS 功能，或者你手动维护了一个 feed.xml，填写其完整访问地址，例如 `/feed.xml` 或 `https://blog.example.com/feed.xml`。填写后会在导航区显示 RSS 图标供访客订阅。

### 部署与 Cloudflare

**deploy_repo**
GitHub 仓库地址，用于将构建产物推送至 GitHub Pages。支持两种格式，二选一填写：

**方式一：SSH（推荐，适合本地已配好密钥的环境）**

```
git@github.com:用户名/仓库名.git
```

例如 `git@github.com:alice/alice.github.io.git`。需确保当前系统已为对应 GitHub 账号配置 SSH 密钥，可在终端执行 `ssh -T git@github.com` 验证。

**方式二：HTTPS + Personal Access Token（适合无法配置 SSH 的环境，如部分 Windows 或 CI）**

```
https://<用户名>:<token>@github.com/用户名/仓库名.git
```

例如 `https://alice:ghp_xxxxxxxxxxxxxxxxxxxx@github.com/alice/alice.github.io.git`。

Token 需在 GitHub → Settings → Developer settings → Personal access tokens 中创建，权限勾选 `repo`（或细粒度权限中的 `Contents: Read and write`）。

> ⚠️ **安全提示**：`deploy_repo` 中以 HTTPS+Token 方式填写的凭证以**明文**形式保存在本地 `content/config.json` 中，该文件不会被构建产物（`public/`）包含、也不会出现在推送的 commit 记录里，但请务必确保 `content/` 目录本身不被提交到公开仓库、不被部署到任何可公网访问的路径，且后台管理面板（默认 `32323` 端口）不要暴露在公网或不受信任的网络环境中，否则存在 Token 泄露风险。

**git_user_name**
推送时使用的 Git 用户名，即 `git config user.name` 的值。填写你的真实姓名或 GitHub 昵称，例如 `Alice Wang`。该信息会出现在每次部署的 commit 记录中。

**git_user_email**
推送时使用的 Git 邮箱，即 `git config user.email` 的值。建议填写与 GitHub 账号绑定的邮箱，例如 `alice@example.com`，否则 commit 记录中的作者信息将无法与 GitHub 账号关联。

**cf_zone_id**
Cloudflare 站点的 Zone ID。登录 Cloudflare 控制台，在对应域名的"概述"页面右侧可以找到 Zone ID，格式为 32 位十六进制字符串，例如 `a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4`。填写后可在后台一键触发全站缓存清理。

**cf_api_token**
Cloudflare API Token。在 Cloudflare 个人资料页"API 令牌"中创建，权限需包含"Cache Purge"。Token 是敏感信息，请勿泄露。在后台界面中该字段以密码输入框形式展示，内容不会明文显示在界面上。

**cf_email**
Cloudflare 账号邮箱（可选）。如果使用的是 Global API Key 而非 API Token，则需要同时填写邮箱和 Key。如果使用的是 API Token（推荐），此字段留空即可。

### 备份

以下字段在后台"备份"标签下方的"云端备份"卡片中填写，详细使用方法见「十一、备份与恢复」章节。

**backup_retention**
本地备份保留份数，默认 `10`。每次创建新备份后会自动清理超出这个份数的旧备份（按创建时间从新到旧保留）。恢复前自动生成的 `prerestore_*` 快照单独最多保留 3 份，不占用这个配额。

**auto_backup_on_deploy**
布尔值，默认关闭。勾选后，每次点击"🚀 推送"成功部署后会自动创建一份本地备份，无需手动操作。

**cloud_backup_url**
云端备份接口的完整地址（`api.php` 所在 URL，需 `https://`），需配合另外部署的 PHP 云端备份系统使用。

**cloud_backup_key**
云端备份接口的访问 Key（32 位，安装云端面板时生成，仅显示一次）。以密码输入框形式展示，通过自定义请求头 `X-Backup-Key` 传输，不出现在 URL 或服务器访问日志中。

### 广告代码

**monetag_tag_code**
Monetag 或其他广告平台的 `<script>` 标签代码。填写后，该代码会被注入到每个页面 `<head>` 标签的末尾。支持完整的脚本标签，例如：

```html
<script src="https://cdn.monetag.com/xxx.js" async></script>
```

如果不需要广告，留空即可。

## 六、内容管理使用教程

内容管理是后台最核心的功能区，涵盖文章与页面的新建、编辑、保存、删除以及触发构建部署的完整工作流。下面按照实际操作顺序逐步讲解。

### 6.1 内容类型说明

系统将内容分为两种类型：

- **文章（posts）**：博客的主要内容单元，有发布日期和标签，按时间倒序排列在首页，每篇文章生成一个独立的 `/archive/UID/` 路径。
- **页面（pages）**：独立的静态页面，不出现在文章列表中，路径直接对应文件名，例如 `about.md` 生成 `/about.html`。适合"关于我"、"版权声明"、"项目展示"等固定内容。

两种类型都支持中文（zh）和英文（en）两个语言版本，互相独立管理。

### 6.2 切换语言和内容类型

在"内容"标签左侧面板顶部有两个下拉菜单：

- 第一个下拉菜单选择语言：`ZH 中文` 或 `EN 英文`
- 第二个下拉菜单选择类型：`文章` 或 `页面`

切换后，左侧文件列表会自动刷新，显示对应目录下的所有 Markdown 文件。中文文章存储在 `content/posts/zh/`，英文文章存储在 `content/posts/en/`，页面同理。

### 6.3 新建文章

点击左侧面板中的"+ 新建"按钮，右侧编辑区会清空，进入新建模式。按以下步骤填写内容：

**第一步：填写标题**
在"文章标题"输入框中输入文章标题，这是必填项。标题会写入 Markdown 文件的 front matter，同时作为页面的 `<title>` 标签内容和首页文章卡片中显示的名称。

**第二步：填写发布日期**
在"发布日期"输入框中填写日期，格式严格为 `YYYY-MM-DD`，例如 `2025-06-13`。系统会根据日期对首页文章列表进行排序，日期越新的文章排在越前面。如果留空，文章的排序位置将不可预测，建议每篇文章都填写日期。

**第三步：填写分类（可选，单个）**
在"分类"输入框中填写一个分类名称，例如 `技术` 或 `生活`，每篇文章只能属于一个分类，可以留空。分类会显示在文章卡片和详情页头部，并用于首页侧栏的"分类"筛选 widget。

**第四步：填写标签**
在"标签"输入框中填写标签，多个标签之间用英文逗号分隔，例如 `技术, Python, 教程`。标签会显示在文章卡片和文章页面上，帮助访客快速了解文章主题，也用于首页侧栏的"标签云"筛选。标签也可以留空。

**第五步：填写封面图（可选）**
在"封面图"输入框中可以直接粘贴任意图片 URL，也可以点击旁边的"📁 素材库"按钮，从已上传到"素材"标签的图片中选一张点击即可自动填入。该字段对应 front matter 的 `cover` 字段，优先级最高，会覆盖全局"文章封面图池"（`post_bg_urls`）和"随机图 API"（`random_img_api`）的设置。留空则按照优先级自动兜底：图池随机抽取 → 随机图 API（带防撞图 uid）→ 全局默认背景图（`bg_url`）→ 以上均未配置时显示扁平化图标占位，不会出现空白或裂图的 `<img>`。

**第六步：撰写正文**
在下方的大型文本区域中输入 Markdown 格式的正文内容。系统支持以下 Markdown 扩展语法：

- **围栏代码块**：使用三个反引号包裹，可指定语言实现语法高亮，例如：
  ```
  ```python
  print("Hello World")
  ```
  ```
- **表格**：标准 GFM 表格语法
- **目录**：在正文中插入 `[TOC]` 会自动生成文章目录
- **代码高亮**：依赖 Pygments，构建时自动处理
- **自动换行**：段落内的单个换行会被保留（nl2br 扩展）
- **图片懒加载**：构建时自动为正文中所有 `<img>` 标签添加 `loading="lazy"` 和 `decoding="async"` 属性

**第七步：保存文件**
点击右侧编辑区顶部的"保存"按钮。首次保存时，系统会自动生成一个唯一标识符（UID，时间戳+两位随机数），该 UID 决定文章的访问路径（`/archive/UID/`）。保存成功后，左侧文件列表会出现新文章，右下角状态栏会显示保存成功提示。

文章以 Markdown 文件形式保存在本地目录中，你也可以直接用文本编辑器打开 `content/posts/zh/` 目录下的 `.md` 文件进行编辑，文件头部的 YAML front matter 格式如下：

```yaml
---
title: 我的第一篇文章
date: 2025-06-13
category: 技术                          # 可选，单个分类，可留空
tags:
  - 技术
  - Python
cover: https://example.com/cover.jpg   # 可选，留空则走全局封面兜底逻辑
uid: "174233xxxxxx"
---

正文内容从这里开始...
```

### 6.4 编辑已有文章

在左侧文件列表中点击任意文章条目，右侧编辑区会自动加载该文章的标题、日期、分类、标签、封面图和正文内容。直接修改后点击"保存"按钮即可覆盖原文件。

文章的 UID 一旦生成便不会改变（除非手动修改 front matter），因此文章的访问 URL 在保存后始终保持不变，不会因为修改标题或内容而失效。

### 6.5 删除文章

在左侧文件列表中点击选中要删除的文章，然后点击右侧编辑区顶部的"删除"按钮。系统会弹出确认对话框，确认后该文章的 Markdown 源文件将被永久删除。注意：删除操作不可撤销，且不会自动触发构建，已删除的文章在下次构建前仍会存在于 `public/` 目录中。

### 6.6 管理独立页面

在左侧面板的类型下拉菜单中选择"页面"，操作方式与文章基本相同（页面同样支持单独填写封面 URL）。页面文件名（不含 `.md` 扩展名）直接决定页面的访问路径，例如文件名为 `about.md` 则访问路径为 `/about.html`。建议使用英文小写字母和连字符命名页面文件，避免中文文件名。

### 6.7 触发构建

内容编辑完成后，需要触发构建才能将 Markdown 源文件转换为可访问的静态 HTML 文件。点击编辑区顶部的"构建"按钮，后台会立即启动构建流程。构建过程是异步执行的，不会阻塞界面操作，且内置互斥锁，构建/部署任务正在进行时再次点击会被拒绝，避免并发冲突。

切换到"日志"标签可以实时查看构建进度和结果。构建完成后终端会输出类似以下内容：

```
[10:32:15] 开始构建...
[10:32:16] ✨ 构建完成！12 篇文章，耗时 1.23s
[10:32:16] 构建完成
```

构建产物输出到项目根目录下的 `public/` 文件夹，包含完整的静态站点文件。

### 6.8 推送至 GitHub Pages

构建完成后，点击"推送"按钮，系统会在 `public/` 目录中初始化 Git 仓库，将所有文件提交，并强制推送到配置项中 `deploy_repo` 指定的 GitHub 仓库 main 分支。每次推送的 commit message 格式为 `update: YYYY-MM-DD HH:MM`。

推送完成后，GitHub Actions 通常会在 1 至 3 分钟内自动完成 Pages 的部署更新。

## 七、素材库使用说明

"素材"标签用于管理上传至 `content/attachments/` 目录的文件（主要是图片）。

点击"上传文件"按钮选择本地文件，支持一次性上传多个文件。上传完成后，文件会出现在下方的素材网格中。点击任意素材条目，上方的"生成的 Markdown / URL 链接"输入框会自动填充该文件对应的 Markdown 图片引用代码，格式如下：

```
![文件名](/attachments/文件名.jpg)
```

点击"复制"按钮将链接复制到剪贴板，然后粘贴到文章正文中即可引用该图片。

构建时，`content/attachments/` 目录中的所有文件会被复制到 `public/attachments/` 目录，使其在最终站点中可以通过 `/attachments/文件名` 路径访问。

如需删除某个素材，点击选中后点击"删除选中"按钮即可从服务器上删除该文件。

## 八、前台主题功能

默认主题（`themes/default/`）除了基础的文章展示外，还内置了一套内容组织与前台交互功能。这些功能全部在 `builder.py`、`themes/default/index.html`、`themes/default/post.html` 以及共享片段 `themes/default/_enhance.html` 中实现，无需额外配置即可使用，部分行为可在后台"配置"标签中开关。

### 8.1 分类 / 标签 / 年月归档

- 首页侧栏提供三个 widget：
  - **分类**：按文章数量从多到少列出所有分类，点击可筛选首页文章列表。
  - **标签云**：展示前 5 个高频标签 + 一个"展开全部"入口，点击进入"全部标签"聚合视图（复用首页整体框架切换显示，不额外生成新页面），列出全站所有标签。
  - **归档**：先显示"年份"列表（每年多少篇），点击某一年后钻取进入该年的"月份"列表，点击月份即筛选出该年月的文章，可通过"返回年份"按钮退回上一级。
- 文章详情页头部会显示该文章的分类和标签徽章，点击会跳转回首页并自动带上对应筛选（例如 `/?tag=python`）。
- 分类 / 标签 / 归档三种筛选可以叠加，出现时会显示"清除筛选"按钮可一键重置。
- 分类/标签/年月数据在构建期（`builder.py`）统计并服务端渲染成真实 HTML，对 SEO 和无 JS 环境友好；筛选是纯前端操作（通过给每张文章卡片打 `data-category` / `data-tags` / `data-year` / `data-month` 属性，JS 只控制卡片显示/隐藏），不需要额外的接口请求。

### 8.2 分页

首页文章列表下方会出现分页条（页码 + 上一页/下一页），每页文章数由配置项 `posts_per_page` 决定（默认 9）。切换页面时会保留当前的分类/标签/归档筛选（先筛选，再分页）。少于一页的文章数量时分页条不显示。

### 8.3 前台"显示设置"面板

点击首页左下角的齿轮图标会弹出一个设置面板，包含：

- **侧边栏位置**：左侧 / 右侧（默认）/ 不显示。
- **主题色**：颜色选择器，选好之后全站强调色（按钮、高亮、渐变、页脚图标等）都会跟着变。
- **恢复默认设置**：一键清掉以上个性化设置，回到网站原本的默认样式（即后台配置的 `theme_color`）。

这些设置存在访客浏览器的 `localStorage` 中，属于访客自己的本地偏好，不会影响其他人看到的效果，也不需要登录或服务器参与。后台"配置"标签中的 `theme_color` 是网站的全局默认值，在构建期就会直接写进页面 CSS，访客第一次打开网站、还没碰过前台调色盘之前看到的就已经是这个默认颜色。

网站目前只保留原有的深色主题，没有提供深浅色模式切换。

### 8.4 动效开关

后台"配置"标签中可以分别独立开关三个动效（对应字段见「五、配置项详解 → 展示与动效」）：

- **鼠标粒子跟随效果**（`enable_particles`）：鼠标移动时拖出一串彩色粒子尾迹。关闭后连 `<canvas>` 节点都不会生成，不占用任何渲染/计算资源。
- **图标旋转动效**（`enable_icon_spin`）：设置齿轮常态缓慢自转，侧边栏分类/标签/归档 widget 标题图标悬停时的旋转反馈。
- **标签 / 分类选中弹跳动画**（`enable_tag_bounce`）：点中的标签/分类会有一个放大回弹的小动画。

三个开关互不影响；侧边栏位置、主题色、分类/标签/归档筛选、分页、回到顶部按钮这些功能性部分不受这几个开关影响，一直可用。页面滚动超过一定距离后，右下角会浮现"回到顶部"按钮，同样属于不受开关影响的基础功能。

### 8.5 文章封面兜底与独立封面

封面图优先级：文章 frontmatter 手动指定的 `cover` > 固定图池（`post_bg_urls`）随机抽取 > 随机图 API（`random_img_api`）> 以上三者都没配置时，显示一个扁平化图标占位（Font Awesome 图标，非 emoji），不会出现空白或裂图的 `<img>`。

在后台"内容"标签编辑文章时，可在"封面图"输入框直接粘贴任意图片 URL，或点击"📁 素材库"按钮从已上传的素材中选一张点击即可自动填入；指定了封面的文章，构建时会固定使用这张图，不会再被随机图池/API 覆盖。

### 8.6 Hero 副标题打字机效果

由 `enable_hero_typing` 配置项控制（默认关闭），开启后首页 Hero 区域的副标题文字会逐字打出，速度是 95～170ms/字左右的随机节奏，标点符号后额外停顿 300～520ms，模拟真人打字；光标是紧跟在最后一个已打出字符后面的闪烁竖线，文字换行时也会跟着走。

### 8.7 页面切换过渡效果

站内导航使用 SPA 式局部刷新（点击站内链接不整页刷新，只替换 `nav` / `hero` / 主体内容 / `footer`），换页瞬间会有一层轻微的毛玻璃模糊过渡遮罩（`backdrop-filter: blur()` + 低透明度深色）。为了不影响这套已经调试好的换页逻辑，8.1～8.6 的新功能全部塞进了独立的共享片段 `_enhance.html`（`index.html` 和 `post.html` 都通过 `{% include '_enhance.html' %}` 引入），设置按钮/粒子画布/回到顶部按钮都是挂在 `<body>` 最外层、不会被 SPA 替换掉的独立节点；页面内容切换后用一个 `MutationObserver` 自动感知变化并重新应用设置/重新绑定筛选。

## 九、追番列表管理

"追番"标签用于维护正在追看的动画列表。该列表数据会通过模板渲染到站点的追番页面或侧边栏中。

点击"+ 添加番剧"打开添加对话框，需填写以下字段：

- **番剧名称**：动画的完整名称，例如 `进击的巨人 最终季`
- **当前集数**：当前已看到的集数，例如 `12`
- **总集数**：该番剧的总集数，例如 `16`，如果还未完结可填写 `?`
- **封面图 URL**：番剧封面图片的直链地址，可从 Bangumi、AniList 等站点获取
- **番剧链接**：指向番剧详情页的 URL

添加后，列表中的条目可以通过"上移"、"下移"按钮调整排列顺序，通过"编辑"按钮修改内容，通过"删除"按钮移除条目。所有修改完成后点击"保存"按钮，数据会立即写入配置文件。重新构建后变更会反映在站点页面中。

## 十、友链管理

"友链"标签用于管理友情链接列表。

点击"+ 添加友链"打开添加对话框，需填写：

## 十一、备份与恢复

"备份"标签（依赖 `backup_manager.py`，与 `main.py` 放在同一目录下即可自动启用，无需额外安装依赖，云端交互仅使用 Python 标准库 `urllib`）提供本地备份和云端备份两部分功能。

### 11.1 备份范围（刻意保持轻量）

- ✅ `content/config.json`（站点配置）
- ✅ `content/posts/**`（全部文章）
- ✅ `content/pages/**`（全部页面）
- ❌ `content/attachments/`（附件）
- ❌ `public/`（构建产物）

只打包纯文本内容是刻意为之：Markdown 文章 + 配置通常只有几十 KB 到几百 KB，整份备份基本能稳定控制在 5MB 以内，即使在网络不稳定（例如跨境访问）的环境下也能快速、可靠地上传/下载，不会遇到大文件传输超时的问题。附件请自行用其他方式做异地备份（例如直接同步 `content/attachments/` 目录，或用网盘/对象存储客户端定期同步，更适合增量同步的场景）。

打包后的 zip 内还会附带一份 `backup_manifest.json`，记录打包时间与包含范围，便于人工核对。

### 11.2 本地备份

- **立即创建备份**：一键打包为 `backup_YYYYMMDD_HHMMSS_xxxxxx.zip`，保存在项目根目录的 `backups/` 文件夹。
- **下载**：下载对应的 zip 到本地电脑。
- **恢复**：从某个本地备份恢复。**恢复前会自动为当前状态创建一份 `prerestore_*.zip` 快照**，如果恢复后发现不对，可以再恢复回这份快照，双重保险。
- **上传 zip 并恢复**：把之前下载到本地电脑的备份文件（或者从云端下载的文件）重新上传回来，可以直接恢复，不需要先放进 `backups/` 目录。
- **删除**：清理不需要的本地备份。
- **本地备份保留份数**（`backup_retention` 配置项，默认 10）：创建新备份时会自动清理超出部分的旧备份，`prerestore_*` 快照单独最多保留 3 份，不占用普通保留份数配额。
- **推送后自动备份**（`auto_backup_on_deploy` 配置项）：勾选后，每次点击"🚀 推送"成功部署后会自动创建一份备份。

恢复操作全程做了 zip 路径穿越（zip slip）防护，且**只允许**恢复 `content/config.json`、`content/posts/`、`content/pages/` 下的条目，即使备份包里混入了其他路径的文件也会被自动忽略，不会覆盖到项目之外的任何文件。

### 11.3 云端备份

云端备份功能需要配合另一个独立的 PHP 云端备份系统一起部署到你自己的服务器/虚拟主机（建议启用 HTTPS）。完成安装向导后会得到一个 API 地址（形如 `https://your-domain.com/backup/api.php`）和一个 32 位的访问安全 Key（仅在安装完成页面显示一次，请务必当场保存）。

将这两项填入"备份"标签下方"云端备份"卡片中的 `cloud_backup_url` / `cloud_backup_key` 字段并保存，即可使用：

- **测试连接**：验证地址和 Key 是否配置正确。
- **上传云端**：把某个本地备份推送到云端保存一份异地副本。
- **刷新云端列表 / 拉取到本地**：查看云端已有哪些备份，并可以随时把某份拉取回本地（拉取后可再执行"恢复"）。
- **删除云端备份**：清理云端存储空间。

访问 Key 通过自定义请求头 `X-Backup-Key` 传输，不会出现在 URL 或服务器访问日志中；云端 `config.json` 中也只保存该 Key 的哈希值，不落地明文，即使云端服务器配置文件意外泄露，也无法反推出可用的 Key。

因为备份体积天然很小（纯文本），整个上传/下载过程通常在一两秒内完成，不容易触发网络不稳定环境下常见的大文件传输超时问题。

### 11.4 备份相关常见问题

**恢复后网站没有更新？**
恢复只是把文件写回磁盘，别忘了恢复后前往"内容"标签检查，并重新点击"🔨 构建"+"🚀 推送"。

**忘记了云端访问 Key 怎么办？**
登录云端管理面板（浏览器访问该 PHP 项目的 `index.php`，用安装时设置的用户名密码），在"安全设置"里点击"重新生成访问 Key"，然后把新 Key 同步更新到本地面板配置中。

**想备份附件怎么办？**
附件（图片等媒体文件）体积通常比文本内容大得多，也更适合用专门的文件同步工具（rsync、网盘客户端、对象存储 CLI 等）做异地备份，而不是塞进这个轻量级的文本备份里——这样既能保持备份体积小、传输快，也能让附件的备份策略更灵活（比如增量同步，而不是每次全量打包）。

## 十二、构建与部署流程

### 构建原理

构建由 `builder.py` 负责执行，完整流程如下：

1. 读取 `content/config.json` 配置文件，并对追番列表、友链、封面图池等字段做类型归一化处理
2. 修复 `site_notice` 公告内容中错误的 `<a src="...">` 写法为 `<a href="...">`
3. 清空 `public/` 目录（保留 `.nojekyll`）
4. 复制 `content/sw.js`（如存在）至 `public/sw.js`
5. 复制 `content/attachments/` 至 `public/attachments/`（中英文目录各一份）
6. 使用最多 6 个线程并发遍历 `content/posts/zh/`、`content/posts/en/` 与 `content/pages/zh/`、`content/pages/en/` 下的所有 `.md` 文件
7. 解析 front matter，为缺失 `uid` 的文件自动生成并写回（使用文本模式写入，兼容当前版本 `python-frontmatter`），渲染 Markdown 正文为 HTML，并按三级优先级（`cover` frontmatter → `post_bg_urls` 图池 → `random_img_api` + uid）解析封面图
8. 使用 Jinja2 模板引擎渲染最终页面，文章输出至 `public/archive/UID/index.html`，页面输出至对应 `.html` 文件
9. 渲染首页 `index.html` 和英文首页 `en/index.html`，首屏背景图自动添加 `fetchpriority="high"` 属性
10. 生成各语言的 `search.json` 搜索索引
11. 生成 `sitemap.xml` 和 `robots.txt`（自动跳过设置了 `noindex` 的页面）
12. 如有自定义域名，生成 `CNAME` 文件

### 部署原理

点击"推送"后，系统先尝试获取任务互斥锁（防止与正在进行的构建/部署冲突），随后在 `public/` 目录执行以下 Git 操作：

```
git init
git config user.name <git_user_name>
git config user.email <git_user_email>
git add .
git commit -m "update: YYYY-MM-DD HH:MM"
git branch -M main
git remote add origin <deploy_repo>
git push -f origin main
```

使用强制推送（`-f`），每次部署都会覆盖远程仓库的历史记录，保持仓库干净，不积累历史 commit。

`deploy_repo` 支持 SSH 地址（`git@github.com:…`）和 HTTPS+Token 地址（`https://用户名:token@github.com/…`），部署脚本直接将配置值作为 remote origin 使用，两种格式均可正常推送，无需额外设置。环境变量 `GIT_TERMINAL_PROMPT=0` 已内置，防止命令行等待交互输入挂起；每条 Git 命令还设有 120 秒超时，超时会在日志中明确提示。

## 十七、常见问题

**Q：点击"推送"后提示"请先配置 GitHub 仓库 SSH 地址"**

请检查"配置"标签中的 `deploy_repo` 字段是否已填写仓库地址。支持两种格式：SSH 格式（`git@github.com:用户名/仓库名.git`）和 HTTPS+Token 格式（`https://用户名:token@github.com/用户名/仓库名.git`）。使用 SSH 时，需确认当前系统已配置好 SSH 密钥，可执行 `ssh -T git@github.com` 验证；使用 HTTPS 时，请确认 Token 具有 `repo` 写权限且未过期。

**Q：文章的 UID 是什么，能不能手动设置？**

UID 是每篇文章的唯一标识符，由时间戳加两位随机数构成（例如 `174233xxxxxx`），在首次保存时自动注入到 Markdown 文件的 front matter 中，决定文章的访问路径 `/archive/UID/`。一旦生成，修改标题或正文不会改变 UID，因此文章 URL 永久稳定。你也可以手动编辑 `.md` 文件的 `uid` 字段来指定自定义值，只要在同语言内唯一即可。

**Q：构建成功但网站没有更新**

构建只是生成了本地的 `public/` 目录，还需要点击"推送"将其发布到 GitHub。如果已推送但仍未更新，请检查 GitHub Pages 的部署来源设置是否指向正确的分支（main），以及 GitHub Actions 是否有报错。如果使用了 Cloudflare 代理，还需要清理 CF 缓存。

**Q：文章顺序不对**

文章按 `date` 字段的字符串值降序排列。请确保日期格式为标准的 `YYYY-MM-DD`，非标准格式（例如 `2025/6/13`）可能导致排序异常。

**Q：构建时提示"找不到 config.json"**

请先运行 `python main.py` 启动后台，系统会在首次启动时自动初始化 `content/config.json`。不要尝试直接运行 `python builder.py`，除非确认 config.json 已存在。

**Q：Cloudflare 缓存清理提示权限错误**

请检查 API Token 的权限范围，确保包含 `Zone.Cache Purge` 权限，且 Token 对应的域名与填写的 Zone ID 匹配。使用全局 API Key 时还需在 `cf_email` 字段填写 Cloudflare 账号邮箱。

**Q：文章图片不显示**

请确认图片路径正确。若图片位于 `content/attachments/`，在 Markdown 中应使用 `/attachments/文件名.jpg` 的绝对路径引用，而非相对路径。也可以使用"素材"标签自动生成的链接代码，避免路径错误。若后台"素材"标签或文章编辑器的"📁 素材库"选图窗口里缩略图/封面预览打不开（提示 404），通常是本地服务缺少对应的静态文件路由，升级到最新版 `main.py`（包含 `/attachments/<filename>` 路由）即可解决。

**Q：配置了文章封面图池或随机图 API，但某些文章背景不是预期效果**

封面图按优先级生效：①单篇文章的"封面图"（frontmatter `cover` 字段）；②全局"文章封面图池"（`post_bg_urls`），每篇文章在构建时随机抽取一张；③"随机图 API"（`random_img_api`），构建时不发起网络请求，而是在 API 地址后拼接每篇文章的唯一 `uid` 参数，由浏览器在访问时各自请求，因此正常情况下不会撞图；④以上均未配置时显示扁平化图标占位。若某篇文章背景与预期不符，优先检查该文章是否单独填写了"封面图"（会覆盖全局设置）；若多篇文章背景重复，且未使用图池，通常是因为这些文章的 `uid` 相同（例如手动批量复制了 `.md` 文件导致 `uid` 未重新生成），正常通过后台新建或保存的文章不会出现此问题。如果希望在构建期就完全确定背景图（而非依赖前端各自请求），改用 `post_bg_urls` 图池。

**Q：推送时日志输出"命令无任何输出却返回失败"**

这通常是两次部署/构建并发冲突导致的 git index.lock 文件竞态，或上次部署未完成就再次点击了推送按钮。稍等片刻后重试一次即可。系统已内置部署互斥锁（`_task_lock`），正常情况下不会出现并发冲突，重试一次基本能解决。

**Q：使用 HTTPS+Token 方式部署，是否安全？**

Token 仅以明文形式保存在本地 `content/config.json` 中，不会包含在构建产物 `public/` 目录或推送的 commit 记录里。但请勿将 `content/` 目录提交到公开仓库，也不要将后台管理面板（默认 `32323` 端口）暴露在公网或不受信任的网络环境中。条件允许的情况下，推荐优先使用 SSH 方式部署以避免明文凭证落盘。

**Q："备份"标签提示"找不到 backup_manager.py"**

请确认 `backup_manager.py` 与 `main.py` 放在同一个项目根目录下，然后重启后台服务。该文件缺失时备份相关功能会被自动禁用，不影响内容管理、构建、部署等其他功能的正常使用。

**Q：自定义主题色后，页脚颜色没有跟着变**

请确认使用的是包含页脚样式修复的最新版 `themes/default/index.html` 和 `post.html`：旧版本中页脚卡片边框和图标 hover 底色写死为固定的粉色 `rgba` 值，没有引用主题色 CSS 变量，导致无论后台默认色还是前台调色盘怎么调都不生效；新版本已改为引用 `var(--accent)` / `var(--border-accent)`，会正确跟随主题色。
